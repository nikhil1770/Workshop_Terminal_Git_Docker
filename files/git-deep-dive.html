§ 01

How Git Actually Works

Most people learn Git as a list of commands to memorise. That works until something goes wrong, and then they're lost. Understanding the model underneath means you can reason about any situation.

The three areas

Every file in a Git project lives in one of three places at any given moment:

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ Working Tree │ │ Staging Area │ │ Repository │ │ │ │ (Index) │ │ (.git folder) │ │ Your actual │───▶│ "What will be │───▶│ Permanent │ │ files on disk │ │ in next commit"│ │ snapshots │ │ │ │ │ │ (commits) │ └─────────────────┘ └─────────────────┘ └─────────────────┘ git add ──────────────────▲ ▲ git commit ────────────────────────────────────────▶

Working Tree — the files you actually see and edit. Staging Area — a holding zone where you build up your next commit. Repository — the permanent history, stored in the hidden .git folder.

What a commit really is

A commit is not a diff (a list of changes). It is a complete snapshot of every tracked file at that moment in time, stored efficiently. Each commit contains: the snapshot, your name/email, a timestamp, a message, and a pointer to its parent commit(s).

📐 Mental Model
Think of commits as a chain of photos. Each photo shows the entire project. Git is smart enough not to store duplicate files — if a file didn't change, it just points to the same file from the previous photo.

The HEAD pointer

HEAD is just a pointer to your current position in history — usually pointing to a branch, which in turn points to the latest commit on that branch. When you make a new commit, the branch pointer moves forward to it, and HEAD follows.

main ──────────────────────────────────────────────▶ commit C3 ▲ HEAD
§ 02

The Core Workflow — In Depth

git status — your best friend

Run git status constantly. It tells you exactly what state every file is in.

git status On branch main Changes to be committed: ← staged (will go in next commit) modified: app.py Changes not staged for commit: ← modified but not staged yet modified: README.md Untracked files: ← new files Git hasn't seen new-file.txt

git add — staging precisely

# Stage one specific file git add filename.txt # Stage everything in current folder git add . # Stage parts of a file interactively (advanced but powerful) git add -p filename.txt

git commit — writing good messages

# Quick one-line commit git commit -m "Add login form validation" # Stage + commit all tracked files in one step git commit -am "Fix typo in README" # Amend the last commit (fix message or add forgotten files) git commit --amend -m "Better message"
✦ Good commit messages
Use the format: verb + what changed. Examples: "Add user authentication", "Fix crash on empty input", "Remove deprecated API calls". Bad: "stuff", "wip", "fix". Your future self will thank you.

git log — reading history

# Full log git log # Compact one-line version (most useful day-to-day) git log --oneline a3f9c12 Add README file b1e2d45 Initial commit # Visual branch graph git log --oneline --graph --all # See what changed in a specific commit git show a3f9c12
§ 03

.gitignore

Some files should never be committed — build artifacts, secrets, editor configs, OS junk. Create a file named .gitignore in your project root and list patterns to ignore.

# Create the file touch .gitignore

A typical .gitignore for a Python/Node project:

# macOS junk .DS_Store # Python __pycache__/ *.pyc .env venv/ # Node.js node_modules/ dist/ # Secrets — NEVER commit these .env *.pem *.key
🔴 Never Commit Secrets
API keys, passwords, and tokens committed to a public GitHub repo are scraped by bots within seconds. Always put credentials in a .env file and add it to .gitignore before your first commit.
§ 04

Branches — Git's Killer Feature

A branch is just a lightweight, movable pointer to a commit. Creating a branch costs almost nothing — it's just creating a new label. This is what makes Git incredibly powerful for experimentation and parallel work.

📐 Why Branches?
Imagine you're writing code and want to try a risky new feature without breaking what already works. Create a branch, experiment freely, and either merge it back when it's ready — or just delete it if the idea was bad. Main stays clean the whole time.

Creating and switching branches

# List all branches (* = current) git branch * main # Create a new branch git branch feature-login # Switch to it git switch feature-login # Shortcut: create AND switch in one command git switch -c feature-login # Go back to main git switch main # Delete a branch (after merging) git branch -d feature-login

What branches look like

C1 ── C2 ── C3 ← main \ C4 ── C5 ← feature-login (you are here) ▲ HEAD

Both branches share the history up to C3. Changes on feature-login don't affect main at all until you merge.

✦ Naming Convention
Name branches after what they do. Common patterns: feature/add-search, fix/login-crash, hotfix/security-patch. Avoid generic names like "test" or "my-branch".
§ 05

Merging

Merging brings two branches back together. There are two types of merge Git can do:

Fast-forward merge (the clean case)

If main hasn't moved since you branched off, Git simply moves the main pointer forward. No new commit is created.

Before: main → C3 ── C4 ── C5 ← feature After: main ─────────────── → C5
git switch main git merge feature-login Fast-forward

Three-way merge (the real case)

If both branches have new commits, Git creates a new merge commit that has two parents.

Before: C3 ── C4 ← main \ C5 ── C6 ← feature After: C3 ── C4 ── M7 ← main (M7 is the merge commit) \ / C5 ── C6

Merge conflicts

A conflict happens when both branches changed the same line of the same file. Git can't decide which to keep — it asks you.

CONFLICT (content): Merge conflict in app.py Automatic merge failed; fix conflicts and then commit the result.

Open the conflicted file — Git marks the problem area:

<<<<<<< HEAD print("Hello from main") ======= print("Hello from feature") >>>>>>> feature-login

Edit the file to keep what you want (delete the markers), then:

git add app.py git commit -m "Merge feature-login into main"
✦ Conflict tip
VS Code will show you conflicts visually with "Accept Current", "Accept Incoming", "Accept Both" buttons. Much easier than editing the markers by hand.
§ 06

Rebasing

Rebase is an alternative to merge. Instead of creating a merge commit, it replays your commits on top of the target branch — as if you had branched off later.

Before: C3 ── C4 ← main \ C5 ── C6 ← feature After rebase: C3 ── C4 ── C5' ── C6' ← feature (replayed on top) ↑ main
git switch feature-login git rebase main
⚠ Golden Rule of Rebasing
Never rebase commits that have already been pushed to a shared remote branch. Rebase rewrites commit history (creates new commit hashes). If others have based work on those commits, their history and yours diverge and everything becomes a nightmare. Rebase only on your own local branches.

Merge vs Rebase — when to use which

SituationUseWhy
Bringing in changes from main to your branchrebaseKeeps your branch history clean
Finishing a feature and bringing it into mainmergePreserves when the feature landed
Branch is public / others use itmergeSafe — never rewrites history
Cleaning up messy local commits before PRrebase -iInteractive rebase lets you squash/edit
§ 07

Remotes

A remote is a version of your repository stored elsewhere — usually on GitHub. origin is the conventional name for the primary remote.

# See your remotes git remote -v origin https://github.com/you/project.git (fetch) origin https://github.com/you/project.git (push) # Add a remote git remote add origin https://github.com/you/project.git # Clone an existing remote repo (creates a local copy) git clone https://github.com/someone/project.git
📐 Remote branches
When you clone a repo, Git creates remote-tracking branches like origin/main. These are read-only snapshots of where the remote branches were the last time you fetched. They update only when you run git fetch or git pull.
§ 08

Pull & Push

Pushing — sending your work up

# Push current branch to origin git push origin main # First-time push of a new branch (-u sets the upstream tracking) git push -u origin feature-login # After -u is set, just this is enough: git push

Fetching vs Pulling

These are commonly confused. The difference matters:

CommandWhat it doesSafe?
git fetchDownloads remote changes into origin/main — but does NOT touch your working filesAlways safe
git pullfetch + merge in one step. Updates your working branch immediatelyUsually fine, but can cause conflicts
# Safe way: fetch first, review, then merge git fetch origin git log --oneline main..origin/main # see what's new git merge origin/main # Quick way: pull directly git pull origin main
✦ Before pushing, always pull first
If teammates have pushed new commits, your push will be rejected. Run git pull first to incorporate their changes, resolve any conflicts, then push.
§ 09

Undoing Things

This is where people panic. Don't. Git almost never permanently deletes anything. Here's a map of how to undo at each stage:

Undo unstaged changes (dangerous)

# Discard changes to a file (CANNOT be undone — gone forever) git restore filename.txt # Discard all unstaged changes git restore .

Undo staging (safe)

# Unstage a file (keeps your changes on disk) git restore --staged filename.txt

Undo commits — three options

CommandWhat it doesHistory
git revert <hash>Creates a new commit that reverses a previous onePreserved ✓
git reset --soft HEAD~1Undoes last commit, keeps changes stagedRewritten
git reset --hard HEAD~1Undoes last commit AND discards all changesRewritten ⚠
# Safest way to undo a public commit git revert a3f9c12 # Undo last commit, keep changes ready to re-commit git reset --soft HEAD~1 # Nuclear option: destroy last commit AND all its changes git reset --hard HEAD~1
🔴 Never force-push to a shared branch
git push --force overwrites the remote's history. If teammates have pulled those commits, everything breaks. Only force-push on your own private branches.
§ 10

Stash — Temporary Shelving

You're in the middle of changes when you suddenly need to switch branches (maybe a teammate needs your help with something urgent). You can't commit half-done work. git stash saves your changes temporarily.

# Save current changes to stash git stash Saved working directory and index state WIP on main # ... do your other work, switch branches, etc ... # Restore your stashed changes git stash pop # See what's in the stash git stash list stash@{0}: WIP on main: a3f9c12 Add README # Stash with a descriptive name git stash push -m "half-done login form"
✦ Stash is for quick saves, not long-term storage
If you're shelving work for more than a few hours, consider creating a WIP commit on a branch instead. Stashes are easy to forget about.
